ON-IT 앱버전 개발일기 9일차: 설정, 프로필, 계정 삭제까지 마무리하기

2026년 6월 8일
프론트엔드 프로젝트
Settings Profile Account Logout SwiftUI

9일차에는 앱의 뒤쪽 흐름을 정리했습니다. 프로젝트와 캔버스가 앱의 중심이라면, 설정 화면은 사용자가 앱을 믿고 쓸 수 있게 만드는 마무리 영역입니다.

설정 화면은 정보와 행동을 나눠서 배치했다

설정 화면에는 프로필, 로그인 방식, 앱 정보, 로그아웃, 계정 삭제가 들어갔습니다. 모두 같은 설정이지만 위험도와 성격이 다르기 때문에 섹션을 나눠 배치했습니다.

struct SettingsView: View {
    @EnvironmentObject var authVM: AuthViewModel
    @State private var showLogoutAlert = false
    @State private var showDeleteAccountAlert = false
    @State private var showEditProfile = false

    var body: some View {
        NavigationStack {
            List {
                profileSection
                loginMethodSection
                appInfoSection
                accountSection
            }
            .navigationTitle("설정")
            .sheet(isPresented: $showEditProfile) {
                EditProfileSheet(authVM: authVM)
            }
        }
    }
}

특히 로그아웃과 계정 삭제는 같은 account 영역에 있지만 시각적으로 다르게 다뤘습니다. 로그아웃은 되돌릴 수 있는 행동이고, 계정 삭제는 되돌리기 어려운 행동이기 때문입니다.

로그인 방식은 사용자가 확인할 수 있게 했다

Google로 로그인했는지, Apple로 로그인했는지 표시하는 것은 작은 정보처럼 보입니다. 하지만 계정 문제를 겪는 사용자는 자신이 어떤 방식으로 가입했는지부터 확인하고 싶어 합니다.

private var loginMethodLabel: String {
    switch authVM.loginMethod {
    case "google": return "Google로 로그인"
    case "apple": return "Apple로 로그인"
    case "guest": return "게스트 모드"
    default: return "로그인 정보 없음"
    }
}

private var loginMethodIcon: String {
    switch authVM.loginMethod {
    case "google": return "g.circle"
    case "apple": return "apple.logo"
    case "guest": return "person.crop.circle.badge.questionmark"
    default: return "person.crop.circle"
    }
}

이 정보는 기술적으로 꼭 필요하지는 않지만, 앱을 배포할 때는 사용자가 자신의 계정 상태를 이해할 수 있어야 한다고 생각했습니다. 특히 OAuth 앱에서는 로그인 방식이 곧 계정 찾기의 단서가 됩니다.

계정 삭제는 실제 사용자와 게스트를 다르게 처리했다

게스트 모드는 개발과 체험을 위해 만든 흐름이라 서버 계정이 없습니다. 그래서 실제 계정 삭제 API를 부르면 안 되고, 게스트 데이터만 일정 시간 숨기는 방식으로 처리했습니다.

func deleteAccount() async {
    if Self.isGuestMode {
        UserDefaults.standard.set(Date(), forKey: "guestDeletedAt")
        logout()
        return
    }

    do {
        try await APIService.shared.deleteAccount()
        logout()
    } catch {
        await MainActor.run {
            self.errorMessage = error.localizedDescription
        }
    }
}

func logout() {
    KeychainService.shared.clearAll()
    UserDefaults.standard.removeObject(forKey: "favoriteProjectIds")
    UserDefaults.standard.removeObject(forKey: "loginMethod")
    isLoggedIn = false
}

내 생각

계정 기능은 성공 케이스보다 예외 케이스가 더 중요했습니다. 게스트와 실제 사용자를 같은 코드로 밀어붙이면 편하지만, 그 편함 때문에 잘못된 API 호출이나 이상한 상태가 생길 수 있습니다. 배포 전에는 이런 부분을 분리하는 게 맞았습니다.

마무리 생각

9일차는 앱을 사용하는 사람의 끝 흐름을 정리한 날이었습니다. 로그인, 프로필 수정, 로그아웃, 계정 삭제까지 이어져야 비로소 앱이 하나의 제품처럼 느껴집니다. 기능의 시작만큼 끝도 중요하다는 생각이 들었습니다.
