ON-IT 앱버전 개발일기 2일차: Google과 Apple 로그인 붙이기

2026년 6월 1일
프론트엔드 백엔드 프로젝트
OAuth Apple Login Google Login ASWebAuthenticationSession JWT

2일차에는 앱의 진짜 입구인 로그인을 붙였습니다. ON-IT은 협업 도구라서 누가 어느 워크스페이스에 들어왔는지가 중요하고, 그래서 로그인 흐름이 흔들리면 뒤의 모든 기능도 흔들립니다.

Google OAuth는 앱 안의 브라우저 세션으로 처리했다

iOS에서 Google 로그인을 직접 구현할 때 가장 신경 쓴 부분은 사용자가 안전하게 인증 페이지를 거치고, 앱으로 돌아왔을 때 code를 안정적으로 받는 것이었습니다. 그래서 `ASWebAuthenticationSession`을 사용했습니다.

func loginWithGoogle() {
    Task {
        let loginURL = try await APIService.shared.getGoogleLoginURL(
            redirectURI: Constants.googleRedirectURI
        )

        authSession = ASWebAuthenticationSession(
            url: loginURL,
            callbackURLScheme: Constants.googleRedirectScheme
        ) { callbackURL, error in
            guard let callbackURL,
                  let code = URLComponents(url: callbackURL, resolvingAgainstBaseURL: false)?
                    .queryItems?
                    .first(where: { $0.name == "code" })?
                    .value else { return }

            Task {
                let response = try await APIService.shared.exchangeGoogleCode(code: code)
                KeychainService.shared.save(
                    accessToken: response.accessToken,
                    refreshToken: response.refreshToken
                )
                self.isLoggedIn = true
            }
        }

        authSession?.presentationContextProvider = self
        authSession?.start()
    }
}

여기서 내가 중요하게 본 건 “로그인 성공”이라는 결과보다 성공 이후 앱 상태가 자연스럽게 바뀌는가였습니다. 토큰 저장, 사용자 이름 파싱, 로그인 방식 저장, 화면 전환이 한 흐름 안에서 끝나야 사용자는 로그인 버튼을 누른 뒤 끊김 없이 프로젝트 목록으로 넘어갈 수 있습니다.

백엔드와 맞추기 위해 파라미터 위치를 명확히 했다

Google code 교환 과정에서는 백엔드 쪽에서 redirect URI와 platform 파라미터가 중복되는 문제가 생길 수 있었습니다. 그래서 앱에서는 POST 요청이지만 body에 한 번 더 넣지 않고, 백엔드가 기대하는 query parameter로만 보냈습니다.

func exchangeGoogleCode(code: String) async throws -> AuthResponse {
    var components = URLComponents(string: "(baseURL)/v1/auth-google/code")!
    components.queryItems = [
        URLQueryItem(name: "code", value: code),
        URLQueryItem(name: "platform", value: "ios"),
        URLQueryItem(name: "redirect_uri", value: Constants.googleRedirectURI)
    ]

    var request = URLRequest(url: components.url!)
    request.httpMethod = "POST"
    request.setValue("application/json", forHTTPHeaderField: "Accept")
    request.setValue("ON-IT-iOS", forHTTPHeaderField: "User-Agent")

    return try await send(request)
}

이 코드는 겉으로 보면 평범한 API 호출이지만, 실제로는 프론트와 백엔드 사이의 약속을 정리한 코드였습니다. 웹에서는 잘 되던 OAuth가 앱에서는 redirect scheme 때문에 조금만 달라도 실패합니다. 그래서 “어디에 어떤 파라미터를 넣는지”를 코드에서 분명하게 드러내는 게 중요했습니다.

내 생각

OAuth 문제는 화면만 보고는 원인을 알기 어렵습니다. 그래서 나는 로그인 코드를 짤 때 예쁘게 추상화하는 것보다, 요청 URL과 파라미터가 눈에 보이게 남는 쪽을 선택했습니다. 나중에 백엔드 로그와 맞춰 볼 때 훨씬 빨랐습니다.

Apple 로그인과 숨겨진 게스트 모드

앱스토어 배포를 생각하면 Apple 로그인도 필요했습니다. Google 로그인만 있으면 iOS 사용자의 기대와도 맞지 않고, 심사 과정에서도 로그인 선택지가 제한적으로 보일 수 있다고 생각했습니다.

func loginWithApple() {
    let provider = ASAuthorizationAppleIDProvider()
    let request = provider.createRequest()
    request.requestedScopes = [.fullName, .email]

    let controller = ASAuthorizationController(authorizationRequests: [request])
    controller.delegate = self
    controller.presentationContextProvider = self
    controller.performRequests()
}

동시에 개발 중에는 서버 계정 없이 화면을 빠르게 확인할 방법도 필요했습니다. 그래서 로그인 화면의 타이틀을 여러 번 누르면 게스트 모드가 열리도록 했습니다. 일반 사용자의 흐름은 해치지 않으면서, 개발 중에는 프로젝트 목록과 캔버스 UI를 빠르게 볼 수 있는 작은 장치였습니다.

마무리 생각

2일차를 지나면서 앱 버전의 로그인은 단순히 “토큰을 받아오는 기능”이 아니라는 걸 느꼈습니다. 백엔드 OAuth 정책, iOS 콜백 스킴, 앱스토어 로그인 요구사항, 개발 중 테스트 편의성까지 한 번에 맞물리는 지점이라서 초반에 시간을 쓰는 게 맞았습니다.
