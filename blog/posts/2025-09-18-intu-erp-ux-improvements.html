오늘은 Intu ERP 시스템의 핵심 기능들을 대폭 개선했습니다. 사용자 경험부터 성능 최적화까지, 개발자로서 정말 뿌듯한 하루였습니다. 특히 작은 UX 개선이 전체 시스템의 사용성을 크게 향상시킨다는 것을 다시 한번 실감할 수 있었습니다.

✨ 프로그램 신청서 UX 대폭 개선

🔍 문제 상황

❌ 기존 문제점

  • 프로그램 생성 시 매번 기본 필드(이름, 이메일, 전화번호)를 수동으로 추가해야 하는 번거로움
  • 사용자가 신청서 작성 시 매번 개인정보를 다시 입력해야 하는 불편함
  • 관리자가 실수로 기본 필드를 삭제하거나 수정할 수 있는 위험성

✅ 해결 방안

  • 프로그램 생성/편집 시 기본 필드 자동 추가
  • 기본 필드 보호 로직으로 수정/삭제 차단
  • 로그인된 사용자 정보로 신청서 자동 완성

🛠️ 구현 과정

1. 자동 기본 필드 추가

// 프로그램 생성 시 기본 필드 자동 추가 const [formFields, setFormFields] = useState<FormField[]>([ { name: 'name', type: 'text', label: '이름', description: '신청자 이름', required: true, placeholder: '이름을 입력해주세요', }, { name: 'email', type: 'email', label: '이메일', description: '연락처 이메일', required: true, placeholder: '이메일을 입력해주세요', }, { name: 'phone', type: 'tel', label: '전화번호', description: '연락처 전화번호', required: true, placeholder: '전화번호를 입력해주세요', } ]);

2. 기본 필드 보호 로직

// 기본 필드 보호 로직 const updateFormField = (index: number, field: Partial<FormField>) => { const currentField = formFields[index]; if (currentField && ['name', 'email', 'phone'].includes(currentField.name)) { toast.error('기본 필드(이름, 이메일, 전화번호)는 수정할 수 없습니다.'); return; } // ... 업데이트 로직 }; const deleteFormField = (index: number) => { const currentField = formFields[index]; if (currentField && ['name', 'email', 'phone'].includes(currentField.name)) { toast.error('기본 필드(이름, 이메일, 전화번호)는 삭제할 수 없습니다.'); return; } // ... 삭제 로직 };

3. 사용자 정보 미리 채우기

// 신청서 작성 시 사용자 정보 자동 완성 const ApplicationForm = ({ programId }: { programId: string }) => { const { user } = useAuth(); const [formData, setFormData] = useState({ name: user?.name || '', email: user?.email || '', phone: user?.phone || '', // ... 기타 필드들 }); // 기본 필드는 비활성화 상태로 표시 return ( <form> {formFields.map((field, index) => ( <div key={index}> {['name', 'email', 'phone'].includes(field.name) ? ( <Input {...field} value={formData[field.name]} disabled className="bg-gray-50" /> ) : ( <Input {...field} value={formData[field.name]} onChange={(e) => setFormData({...formData, [field.name]: e.target.value})} /> )} {['name', 'email', 'phone'].includes(field.name) && ( <span className="text-sm text-gray-500">기본 필드 (수정 불가)</span> )} </div> ))} </form> ); };

🎉 구현된 기능

  • 자동 기본 필드 추가: 프로그램 생성/편집 시 이름, 이메일, 전화번호 필드 자동 생성
  • 필드 보호 로직: 기본 필드는 수정/삭제 불가능하도록 완전 차단
  • 사용자 정보 미리 채우기: 로그인된 사용자의 정보로 신청서 자동 완성
  • 수정 불가 처리: 기본 필드는 비활성화 상태로 표시
  • 직관적인 UI: "기본 필드 (삭제 불가)" 안내 메시지 표시

🎨 관리자 UI/UX 대폭 개선

🔍 문제 상황

❌ 기존 문제점

  • 직관적이지 않은 네비게이션 구조
  • 관리자 권한에 따른 메뉴 표시 로직 부족

✅ 해결 방안

  • 우측 상단 드롭다운에 "프로그램 보기" 메뉴 추가
  • 아이콘 기반 직관적인 UI
  • 권한 기반 조건부 렌더링

🚨 신고 시스템 완전 구현

🔍 문제 상황

❌ 기존 문제점

  • 회원들의 신고 이력을 관리자가 파악하기 어려움
  • 신고된 회원을 식별하기 위한 시각적 표시 부족
  • 신고 이력이 있는 회원과 없는 회원을 구분하기 어려움

✅ 해결 방안

  • 전체 회원 조회 시 신고 이력 배지 표시
  • 프로그램 신청자 목록에서도 신고 이력 확인 가능
  • 스마트 표시: 신고 0건인 경우 표시하지 않도록 개선

🛠️ 구현 과정

1. 백엔드 쿼리 최적화

🎉 구현된 기능

  • 신고 이력 표시: 전체 회원 조회 시 신고 이력 배지 표시
  • 프로그램 신청자 신고 이력: 신청자 목록에서도 신고 이력 확인 가능
  • 스마트 표시: 신고 0건인 경우 표시하지 않도록 개선
  • 시각적 구분: 주황색 배지로 신고 이력 명확히 표시

⚡ 백엔드 쿼리 최적화

🔍 문제 상황

❌ 기존 문제점

  • GROUP BY와 COUNT를 함께 사용할 때의 쿼리 성능 문제
  • getManyAndCount() 메서드의 한계
  • 복잡한 조인 쿼리에서의 데이터 정확성 문제

✅ 해결 방안

  • getRawAndEntities() 사용으로 성능 향상
  • 별도 카운트 쿼리로 정확한 페이지네이션
  • GROUP BY와 COUNT 조합에서의 데이터 정확성 보장

🛠️ 구현 과정

쿼리 최적화

// 기존 방식 (문제) const [users, total] = await queryBuilder.getManyAndCount(); // 개선된 방식 (해결) const { entities: users, raw } = await queryBuilder.getRawAndEntities(); const totalCount = await totalQuery.getCount(); // 결과 매핑 const usersWithReportCount = users.map((user, index) => ({ ...user, reportCount: parseInt(raw[index].reportCount) || 0 }));

🎉 구현된 기능

  • 쿼리 성능 최적화: getRawAndEntities() 사용으로 성능 향상
  • 정확한 카운팅: 별도 카운트 쿼리로 정확한 페이지네이션
  • 데이터 무결성: GROUP BY와 COUNT 조합에서의 데이터 정확성 보장

🔧 기술적 도전과 해결

1. 복잡한 쿼리 최적화

// 신고 이력을 포함한 사용자 조회 쿼리 SELECT user.*, COUNT(DISTINCT user_reports.id) as reportCount FROM users user LEFT JOIN user_reports ON user_reports.reportedUserId = user.id GROUP BY user.id

2. React 조건부 렌더링 개선

// 기존 방식 (문제) {user.reportCount && user.reportCount > 0 && ( <Badge>신고 {user.reportCount}건</Badge> )} // 개선된 방식 (해결) {user.reportCount && user.reportCount > 0 ? ( <Badge>신고 {user.reportCount}건</Badge> ) : null}
