ON-IT 앱버전 개발일기 7일차: STOMP 웹소켓을 Swift에서 직접 다루기

2026년 6월 6일
프론트엔드 백엔드 학습
WebSocket STOMP SockJS 실시간협업 Swift

7일차에는 실시간 협업의 핵심인 웹소켓을 붙였습니다. 웹에서는 라이브러리가 많은 일을 대신해줬지만, iOS에서는 서버가 쓰는 SockJS/STOMP 흐름을 앱에서 직접 이해해야 했습니다.

연결은 단순한 wss가 아니라 STOMP 세션이었다

서버의 웹소켓 엔드포인트는 단순 JSON 소켓이 아니라 STOMP 프로토콜을 사용했습니다. 그래서 연결 후 바로 메시지를 보내는 것이 아니라, 먼저 CONNECT 프레임을 보내고 구독을 등록해야 했습니다.

func connect(workspaceId: Int, token: String) {
    let url = URL(string: "(Constants.socketBaseURL)/ws/websocket")!
    var request = URLRequest(url: url)
    request.setValue("Bearer (token)", forHTTPHeaderField: "Authorization")

    task = URLSession.shared.webSocketTask(with: request)
    task?.resume()

    DispatchQueue.main.asyncAfter(deadline: .now() + 0.5) {
        self.sendFrame("""
        CONNECT
        accept-version:1.2
        heart-beat:10000,10000
        Authorization:Bearer (token)
        workspaceId:(workspaceId)
        userId:(self.userIdFromJWT(token) ?? 0)

        \u{0}
        """)
    }

    receiveLoop()
}

이 코드를 쓰면서 느낀 건, 실시간 기능은 “연결됐다”와 “쓸 수 있다”가 다르다는 점입니다. 웹소켓 연결 자체는 열려도 STOMP CONNECT가 완료되지 않으면 구독과 발행이 안정적으로 동작하지 않습니다.

SockJS 프레임을 파싱해야 했다

서버가 SockJS 형식으로 프레임을 보내기 때문에 앱은 `o`, `h`, `a[...]`, `c[...]` 같은 프레임을 구분해야 했습니다. 이걸 처리하지 않으면 실제 STOMP 메시지까지 도달하지 못합니다.

private func handleRawMessage(_ raw: String) {
    if raw == "o" || raw == "h" { return }

    if raw.hasPrefix("a[") {
        let frames = parseSockJSArray(raw)
        frames.forEach(handleStompFrame)
        return
    }

    if raw.hasPrefix("c[") {
        isConnected = false
        reconnect()
        return
    }

    handleStompFrame(raw)
}

처음에는 메시지가 안 오는 것처럼 보였지만, 실제로는 메시지가 오고 있었고 앱이 그 포장을 풀지 못하고 있었습니다. 그래서 네트워크 문제라고 단정하지 않고, 서버가 보내는 실제 문자열을 기준으로 파서를 만들었습니다.

내 생각

실시간 통신은 추상화가 잘 되어 있으면 편하지만, 문제가 생기면 결국 프레임 단위로 내려가야 합니다. 이번에는 “라이브러리가 알아서 해주겠지”보다 서버 프로토콜을 직접 읽는 쪽이 더 빠른 해결책이었습니다.

채팅, 노트, 사용자 상태를 같은 연결에서 구독했다

ON-IT의 실시간 기능은 채팅만이 아닙니다. 노트 변경, 캔버스 이벤트, 사용자 접속 상태도 모두 협업 경험의 일부입니다. 그래서 연결이 완료되면 여러 topic을 구독했습니다.

private func subscribeAll(workspaceId: Int) {
    subscribe(destination: "/topic/workspace/(workspaceId)/messages")
    subscribe(destination: "/topic/workspace/(workspaceId)/ideas")
    subscribe(destination: "/topic/workspace/(workspaceId)/canvas")
    subscribe(destination: "/topic/workspace/(workspaceId)/users")

    flushPendingIdeaUpdates()
}

func send(_ text: String, workspaceId: Int) {
    sendFrame("""
    SEND
    destination:/app/chat/message
    content-type:application/json

    {"workspaceId":(workspaceId),"message":"(text)"}\u{0}
    """)
}

아이디어 업데이트는 연결이 완전히 준비되기 전에 발생할 수도 있어서 pending queue를 두었습니다. 사용자가 노트를 만들었는데 웹소켓이 아직 CONNECTED 상태가 아니라고 업데이트를 잃어버리면 협업 도구로서는 치명적이라고 생각했습니다.

마무리 생각

7일차는 기술적으로 가장 거친 부분 중 하나였습니다. 하지만 STOMP와 SockJS를 직접 다루면서 앱이 서버의 실시간 협업 구조를 제대로 이해하게 됐고, 이후 채팅과 멤버 상태를 붙일 수 있는 기반이 만들어졌습니다.
