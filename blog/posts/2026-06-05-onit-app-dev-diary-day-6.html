ON-IT 앱버전 개발일기 6일차: 노트 CRUD와 낙관적 업데이트

2026년 6월 5일
프론트엔드 백엔드 프로젝트
CRUD Optimistic Update REST API WebSocket Swift Concurrency

6일차에는 캔버스가 보기만 하는 공간이 아니라 실제로 편집되는 공간이 되도록 만들었습니다. 노트 생성, 이동, 내용 수정, 삭제를 서버 API와 연결하고, 사용자가 기다리지 않도록 낙관적 업데이트를 넣었습니다.

생성은 먼저 화면에 보이게 했다

노트를 만들 때 서버 응답을 기다린 뒤 화면에 추가하면 안정적이지만, 모바일에서는 반응이 늦게 느껴집니다. 그래서 임시 ID를 가진 노트를 먼저 추가하고, 서버 응답이 오면 실제 ID로 바꾸는 방식을 택했습니다.

func addNote(content: String, x: Double, y: Double) async {
    let tempId = -Int(Date().timeIntervalSince1970 * 1000)
    let tempNote = Note(
        id: tempId,
        workspaceId: workspaceId,
        content: content,
        positionX: x,
        positionY: y,
        patchSizeX: 500,
        patchSizeY: 400
    )

    notes.append(tempNote)

    do {
        let created = try await APIService.shared.createNote(
            workspaceId: workspaceId,
            content: content,
            positionX: x,
            positionY: y
        )

        if let index = notes.firstIndex(where: { $0.id == tempId }) {
            notes[index] = created
        }

        socketService.sendIdeaUpdate(workspaceId: workspaceId, note: created)
    } catch {
        notes.removeAll { $0.id == tempId }
        errorMessage = error.localizedDescription
    }
}

임시 ID를 음수로 만든 건 서버에서 내려오는 실제 ID와 충돌하지 않게 하기 위해서입니다. 사용자는 탭하자마자 노트가 생겼다고 느끼고, 실패했을 때만 다시 제거하면 됩니다. 협업 도구에서는 이런 작은 즉시성이 꽤 중요하다고 생각했습니다.

수정 API는 서버가 기대하는 형태에 맞췄다

노트 수정은 단순히 content만 바꾸는 기능이 아닙니다. 위치와 크기도 함께 서버에 남아야 다른 사용자가 같은 캔버스를 볼 수 있습니다. 그래서 수정 요청에는 workspaceId, content, position, patch size를 함께 담았습니다.

func updateNote(_ note: Note) async throws -> Note {
    let url = URL(string: "(baseURL)/v1/ideas/(note.id)")!
    var request = authorizedRequest(url: url)
    request.httpMethod = "PUT"
    request.setValue("application/json", forHTTPHeaderField: "Content-Type")

    let body: [String: Any] = [
        "workspaceId": note.workspaceId ?? 0,
        "content": note.content,
        "positionX": note.positionX,
        "positionY": note.positionY,
        "patchSizeX": note.patchSizeX ?? 500,
        "patchSizeY": note.patchSizeY ?? 400
    ]

    request.httpBody = try JSONSerialization.data(withJSONObject: body)
    return try await send(request)
}

처음에는 PATCH가 더 자연스러워 보였지만, 실제 서버 계약과 맞추는 것이 더 중요했습니다. 앱 쪽에서 “REST 원칙상 이게 더 예쁘다”라고 버티는 것보다, 지금 백엔드가 안정적으로 처리하는 방식에 맞춰 사용자 기능을 완성하는 게 맞다고 판단했습니다.

편집 결과는 웹소켓으로도 알려야 했다

REST API는 서버 저장을 위한 통로이고, 웹소켓은 다른 참여자에게 변화가 있었다고 알려주는 통로입니다. 그래서 노트를 수정하거나 이동한 뒤에는 REST 요청만 끝내지 않고, 아이디어 업데이트 메시지도 보냈습니다.

func moveNote(_ note: Note, to point: CGPoint) async {
    guard let index = notes.firstIndex(where: { $0.id == note.id }) else { return }

    notes[index].positionX = point.x
    notes[index].positionY = point.y

    do {
        let updated = try await APIService.shared.updateNote(notes[index])
        notes[index] = updated
        socketService.sendIdeaUpdate(workspaceId: workspaceId, note: updated)
    } catch {
        errorMessage = error.localizedDescription
    }
}

내 생각

협업 앱에서 저장과 동기화는 같은 기능처럼 보이지만 실제로는 역할이 다릅니다. 서버에 저장만 하면 내 화면은 맞지만 다른 사람 화면은 늦고, 웹소켓만 보내면 새로 들어온 사용자가 상태를 잃습니다. 두 길을 같이 가져가야 했습니다.

마무리 생각

6일차에는 ON-IT 캔버스가 “움직이는 화면”에서 “함께 편집되는 작업 공간”으로 바뀌었습니다. 특히 낙관적 업데이트를 넣으면서, 앱 개발은 서버의 정확성과 사용자의 체감 속도 사이에서 계속 균형을 잡는 일이라는 걸 느꼈습니다.
