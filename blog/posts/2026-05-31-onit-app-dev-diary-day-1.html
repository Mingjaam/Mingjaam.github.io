ON-IT 앱버전 개발일기 1일차: 웹 서비스를 iOS 앱의 뼈대로 옮기기

2026년 5월 31일
프론트엔드 프로젝트
ON-IT SwiftUI iOS 앱구조 Keychain

웹에서 돌아가던 ON-IT을 앱으로 옮기기 시작하면서 가장 먼저 한 일은 기능을 많이 넣는 것이 아니라, 앱이 어떤 상태에서 어떤 화면을 보여줄지 정하는 일이었습니다.

처음에는 화면보다 상태가 먼저였다

앱 버전은 웹처럼 주소를 직접 치고 들어가는 구조가 아니기 때문에, 실행하자마자 로그인 상태를 판단하고 사용자가 갈 수 있는 화면을 제한해야 했습니다. 그래서 첫날에는 캔버스나 채팅보다 앱의 입구를 먼저 만들었습니다.

struct ContentView: View {
    @EnvironmentObject var authVM: AuthViewModel

    var body: some View {
        Group {
            if authVM.isLoggedIn {
                MainTabView()
            } else {
                LoginView(authVM: authVM)
            }
        }
    }
}

struct MainTabView: View {
    var body: some View {
        TabView {
            ProjectListView()
                .tabItem { Label("프로젝트", systemImage: "folder") }

            SettingsView()
                .tabItem { Label("설정", systemImage: "gearshape") }
        }
    }
}

이 구조를 고른 이유는 단순합니다. 로그인 여부가 앱 전체의 최상위 조건이기 때문입니다. 프로젝트 목록, 캔버스, 설정은 모두 토큰이 있어야 의미가 있고, 토큰이 없으면 사용자는 로그인 화면으로 돌아가야 합니다. 이 판단을 각 화면마다 흩어 놓으면 나중에 로그아웃, 토큰 만료, 계정 삭제 같은 기능을 붙일 때 코드가 금방 복잡해질 것 같았습니다.

내가 처음 잡은 기준

앱 구조를 만들 때는 멋있는 화면보다 “틀렸을 때 어디로 돌아갈 수 있는가”를 먼저 생각했습니다. 인증 상태 하나만 믿고 화면이 갈라지게 만들면, 나중에 기능이 늘어도 길을 잃지 않을 것 같았습니다.

환경값은 Constants로 모으기

웹에서 쓰던 API 주소와 앱에서 쓰는 OAuth 콜백 주소가 섞이기 시작하면 디버깅이 어려워집니다. 그래서 앱에서 공통으로 쓰는 주소는 한 곳에 모았습니다.

enum Constants {
    static let apiBaseURL = "https://on-it.kro.kr/api"
    static let socketBaseURL = "https://on-it.kro.kr/api"

    static let googleRedirectScheme = "com.googleusercontent.apps..."
    static let googleRedirectURI = "com.googleusercontent.apps...:/oauthredirect"
}

처음에는 화면마다 문자열을 바로 넣어도 빨리 만들 수 있습니다. 하지만 iOS 앱은 로그인 콜백, REST API, 웹소켓처럼 같은 도메인을 여러 레이어에서 씁니다. 하루 이틀 지나면 “이 주소가 어디서 온 거지?”라는 질문이 생길 게 뻔해서, 초반부터 상수로 모아두는 편이 더 낫다고 봤습니다.

토큰은 UserDefaults가 아니라 Keychain으로

로그인 토큰은 앱의 핵심 보안 정보입니다. 그래서 단순 설정값처럼 `UserDefaults`에 넣지 않고 Keychain을 따로 감싼 서비스를 만들었습니다.

final class KeychainService {
    static let shared = KeychainService()

    func save(accessToken: String, refreshToken: String) {
        save(key: "accessToken", value: accessToken)
        save(key: "refreshToken", value: refreshToken)
    }

    var accessToken: String? {
        read(key: "accessToken")
    }

    var isLoggedIn: Bool {
        accessToken != nil
    }

    func clearAll() {
        delete(key: "accessToken")
        delete(key: "refreshToken")
    }
}

이렇게 감싸두면 화면 쪽에서는 Keychain의 세부 API를 몰라도 됩니다. 인증 흐름에서는 `isLoggedIn`만 보고, 로그아웃할 때는 `clearAll()`만 호출하면 됩니다. 내가 앱 버전에서 가장 피하고 싶었던 건 보안 저장소 코드가 View 곳곳에 새는 것이었습니다.

마무리 생각

1일차의 결과물은 눈에 보이는 기능이 많지는 않았지만, 앱이 흔들리지 않게 서 있을 뼈대를 만든 날이었습니다. ON-IT을 앱으로 옮기려면 웹 기능을 그대로 복사하는 게 아니라, 모바일 앱의 생명주기와 인증 흐름에 맞춰 다시 정리해야 한다는 걸 처음부터 느꼈습니다.
